# 시그널 핸들러 안전성 (Signal Handler Safety)

## 정의

- **시그널(signal)**: OS가 프로세스에게 보내는 비동기 알림. 예: 터미널에서 `Ctrl+C` → `SIGINT`
- **시그널 핸들러**: 특정 시그널이 도착했을 때 실행되도록 등록해둔 함수 (`signal(SIGINT, handler)`로 등록)

## 원리

시그널은 프로그램이 어느 지점을 실행 중이든 상관없이 도착할 수 있다(비동기적). 시그널이 도착하면 OS는 현재 실행 흐름을 일시정지시키고 핸들러를 실행한 뒤, 끝나면 원래 실행하던 지점으로 복귀한다. 즉 핸들러는 별도의 실행 흐름이 언제 어디서든 끼어드는 것과 비슷하다.

이 비동기성 때문에 두 가지 문제가 생길 수 있다.

### 1) 컴파일러 최적화와 `volatile`

컴파일러는 최적화 시 특정 변수가 함수 내에서 바뀌는지 분석해서, 안 바뀐다고 판단되면 값을 레지스터에 캐싱하고 메모리를 다시 읽지 않는다. 그런데 시그널 핸들러가 전역 변수를 바꾸는 것은 컴파일러의 일반적인 제어 흐름 분석 범위 밖이라, 컴파일러가 이를 감지하지 못할 수 있다.

```c
int done = 0;
void handler(int sig) { done = 1; }
int main() {
    signal(SIGINT, handler);
    while (!done) sleep(1);   // 최적화 빌드에서 done을 캐싱해버리면 무한루프
}
```

- **해결**: `volatile sig_atomic_t done;`으로 선언
  - `volatile`: "이 변수는 컴파일러가 모르는 경로로 바뀔 수 있으니 매번 메모리에서 다시 읽어라"라고 지시
  - `sig_atomic_t`: 표준이 시그널 핸들러-메인 코드 간 안전한 접근을 명시적으로 보장하는 타입

### 2) Async-signal-safe 함수

`printf`, `malloc` 같은 표준 라이브러리 함수는 내부적으로 상태(버퍼, 락)를 유지한다. 메인 코드가 이런 함수를 실행하는 도중(내부 상태를 만지는 도중)에 시그널이 도착해서 핸들러가 같은 함수를 다시 호출하면, 같은 내부 상태에 재진입하게 된다 → 데이터 손상, 최악의 경우 데드락으로 이어질 수 있다.

- **async-signal-safe 함수**: 시그널 핸들러 안에서 호출해도 안전하다고 표준이 보장하는 함수 목록. `printf`는 이 목록에 없음
- **해결**: `write()` 같은 저수준 시스템콜 사용 (버퍼링/락 없이 커널에 직접 전달되므로 async-signal-safe 목록에 포함됨)

## 이유 (왜 중요한가)

- 타이밍에 의존적인 버그라서 로컬 테스트에서 우연히 재현이 안 될 수 있다 (시그널이 마침 안전한 타이밍에만 도착하면 정상 동작하는 것처럼 보임) → 발견이 어려움
- `-Wall -Wextra` 같은 일반 컴파일러 경고로는 잡히지 않음

## 핵심 차이/포인트

- `volatile` 단독으로도 캐싱 문제는 해결되지만, 표준이 명시적으로 보장하는 타입은 `sig_atomic_t`이므로 관례적으로 `volatile sig_atomic_t`를 세트로 씀
- **발견 방법**: CERT C 시큐어 코딩 가이드라인의 `SIG30-C`(핸들러 안에서는 async-signal-safe 함수만 호출), `SIG31-C`(공유 객체는 보호 없이 핸들러 안에서 접근 금지) 규칙 참고. `clang-tidy`의 `cert-sig30-c`, `cppcheck`, Clang Static Analyzer 같은 정적 분석 툴이 이 규칙들을 체크해줌
