# 파일 디스크립터(fd)와 소켓(socket)

## 정의

- **파일 디스크립터(fd)**: (커널이 관리하는) 리소스를 프로세스가 참조할 수 있도록 내주는 정수 핸들(키). 상위 메커니즘 개념 — 파일, 파이프, 소켓 등 커널이 다루는 리소스라면 종류를 가리지 않고 이 핸들 하나로 프로세스에 노출됨
	(핸들 뜻 : 실제 데이터/자원의 내부 구조는 노출하지 않고 접근할 수 있는 참조값)
- **소켓(socket)**: fd가 가리킬 수 있는 리소스 타입 중 하나. 네트워크 통신(또는 로컬 IPC)을 위한 자료구조. `socket()` 시스템콜로 생성됨

## 원리

프로세스마다 fd 테이블이 있고, fd(정수)는 그 테이블의 인덱스임. 이 인덱스는 커널 내부의 실제 자료구조(파일 struct, 소켓 struct 등)를 가리킴.

```
프로세스의 fd 테이블
┌─────┬──────────────────────────┐
│ fd  │ 커널 내부 객체 (리소스 타입) │
├─────┼──────────────────────────┤
│  0  │ stdin (파일)              │
│  1  │ stdout (파일)             │
│  2  │ stderr (파일)             │
│  3  │ file_struct (data.txt)   │
│  4  │ socket_struct (TCP)      │
└─────┴──────────────────────────┘
```

fd 자체는 그냥 정수라서 그것만 보고는 파일인지 소켓인지 구분이 안 됨. 어떤 리소스 타입인지는 커널이 fd 테이블 내부에서 관리함.

### 같은 시스템콜, 다른 동작

```c
int fd1 = open("data.txt", O_RDONLY);       // fd가 가리키는 리소스 타입: 파일
int fd2 = socket(AF_INET, SOCK_STREAM, 0);  // fd가 가리키는 리소스 타입: 소켓

write(fd1, buf, 100);   // 커널: "fd1 → 파일" → 디스크 I/O (VFS → 파일시스템 드라이버)
write(fd2, buf, 100);   // 커널: "fd2 → 소켓" → TCP 스택 태워서 네트워크 전송
```

`write()`라는 시스템콜은 리소스 타입과 무관하게 동일하지만, 커널이 fd 테이블에서 실제 가리키는 리소스 타입을 보고 완전히 다른 코드 경로(디스크 I/O vs 네트워크 스택)로 분기함. 이게 유닉스 "everything is a file" 추상화의 작동 방식 — 인터페이스는 통일, 내부 구현은 리소스 타입별로 분기.

소켓 전용 시스템콜(`connect()`, `bind()`, `listen()`, `accept()`)은 fd가 가리키는 리소스가 진짜 소켓 타입인지 커널이 먼저 체크함:

```c
connect(fd1, ...);  // ENOTSOCK 에러 (fd1이 가리키는 건 파일이라 소켓 동작 불가)
connect(fd2, ...);  // 정상 동작
```

## 핵심 차이/포인트

- fd는 리소스 타입과 무관한 정수 핸들임. — fd라는 숫자 자체엔 타입 정보가 없음
- 소켓은 fd가 가리킬 수 있는 여러 리소스 타입 중 하나일 뿐 (파일, 파이프도 마찬가지로 fd로 다뤄짐)
- 어떤 타입인지 판단하고 그에 맞게 동작을 분기하는 건 전부 커널의  몫
