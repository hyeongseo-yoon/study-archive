# 서버의 요청 처리 방식

## 정의

- 서버 = 클라이언트 요청을 기다리다가(listen), 요청이 오면 처리해서 응답을 돌려주는 프로세스 
	-> **요청-응답(request-response) 모델**

## 요청 대기 중 CPU를 낭비하지 않는 원리

- 잘못된 직관: `while(1) { 요청 왔는지 체크 }`처럼 무한루프로 계속 확인하는 방식 → **busy waiting**. CPU를 낭비하고 다른 프로세스의 스케줄링 기회까지 뺏음
- 실제 메커니즘: **blocking system call** (`accept()`, `read()` 등)
  1. 프로세스가 blocking syscall 호출 → **WAITING(blocked)** 상태로 전환, CPU에서 내려감
  2. 클라이언트 요청 패킷이 NIC(네트워크 카드)에 도착 → **하드웨어 인터럽트** 발생
  3. 인터럽트 핸들러(커널)가 실행되며 네트워크 스택이 데이터 처리, 어느 blocked 프로세스가 기다리던 데이터인지 판단
  4. 해당 프로세스를 **READY** 상태로 전환
  5. 스케줄러가 CPU를 배정 → **RUNNING**, syscall이 리턴되며 요청 처리 시작
- 즉 "체크"는 프로세스가 스스로 폴링하는 게 아니라, 하드웨어 인터럽트 + 커널이 대신 해주는 것

## 동시 접속 처리: C10K 문제와 아키텍처의 발전

- **C10K 문제**(1999년): 서버 한 대가 클라이언트 수만 개의 동시 연결(concurrent connection)을 어떻게 효율적으로 처리할지에 대한 문제

| 방식 | 설명 | 장점 | 한계 |
|---|---|---|---|
| process-per-connection | 연결마다 프로세스 하나 (옛날 Apache prefork) | 구현 단순, 프로세스 간 격리 | 생성/컨텍스트 스위치 비용 큼, 메모리 많이 씀 |
| thread-per-connection | 연결마다 스레드 하나 | 프로세스보다 가벼움(주소공간 공유) | 여전히 연결 수 = 스레드 수 → 수만 개면 스택 메모리/컨텍스트 스위치 부담 재발생 |
| event-driven (`select`/`poll`/`epoll`) | 적은 수의 스레드가 다수의 소켓을 동시 감시 | 워커 수가 연결 수와 무관(I/O-bound에 유리) | 하나의 요청이 CPU를 오래 잡으면(CPU-bound), 그 스레드에 물린 나머지 연결이 전부 지연됨 (선점형 스케줄링이 아니므로) |

## 실무: 하이브리드 구조

- **nginx**: CPU 코어 수만큼 워커 프로세스를 띄우고, 워커마다 epoll 이벤트 루프 실행 → 멀티프로세스 + 이벤트 루프
- **Node.js**: 메인 이벤트 루프는 하나, CPU 무거운 작업/blocking I/O는 별도 스레드 풀(libuv)에 위임
- 결론: I/O-bound 작업은 event-driven이 유리하고 CPU-bound 작업은 여전히 별도 프로세스/스레드로 분리해서 처리하는 하이브리드로 수렴함
