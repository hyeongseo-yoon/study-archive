## 0. 터미널의 기초 개념

### 정의

- **터미널(Terminal)**: 사용자와 프로그램(주로 셸) 사이를 이어주는 **입출력 인터페이스**. 키 입력을 받아 셸에 전달하고, 셸이 돌려주는 결과를 화면에 표시하는 역할만 함 — 명령어 자체를 해석하지는 않음. 그냥 "창문" 역할.
- **셸(Shell, bash/zsh 등)**: 터미널 안에서 실제로 명령어를 해석하고 실행하는 프로그램. 창문 뒤에서 일하는 "안내원/일꾼" 역할.
- 터미널과 셸은 서로 다른 개념이며, 셸에는 bash/zsh 등 여러 종류가 있고 문법이 조금씩 다르다.
- 현재 사용 중인 셸 확인: `echo $SHELL`

### 원리

- GUI는 마우스로 아이콘/폴더를 클릭해서 조작하는 방식.
- CLI(터미널)는 키보드로 명령어를 쳐서 조작하는 방식.
- 예: 폴더 100개 안의 파일을 각각 지워야 할 때, GUI는 더블클릭 → 우클릭 → 삭제를 100번 반복해야 하지만, CLI는 명령어 한 줄로 한 번에 처리 가능.
- **명령 처리 흐름**: 사용자 → (터미널이 입력을 받아 전달) → 셸이 명령어 해석 → 실제 프로그램 실행 → 결과가 다시 터미널을 통해 출력됨. 즉 터미널=입출력 인터페이스, 셸=해석/실행 담당이며 이 둘은 별개의 프로그램.

### 이유 (왜 CLI를 쓰는가)

- **자동화·반복**: 명령어는 정해진 절차(레시피)라서 여러 번 반복해도 항상 같은 동작을 보장.
- **재현성**: 마우스로 한 작업은 기록이 남지 않지만, 명령어는 그대로 복사해 남에게 전달하거나 스크립트로 저장해두면 언제든 동일하게 재현 가능.

###  명령어 예시

| 명령어           | 기능                                |
| ------------- | --------------------------------- |
| `echo hello`  | "hello" 문자열을 그대로 화면에 출력           |
| `whoami`      | 현재 로그인한 사용자 이름(계정명) 출력            |
| `date`        | 현재 날짜/시간 출력                       |
| `echo $SHELL` | 현재 사용 중인 셸 경로 확인 (예: `/bin/bash`) |
|               |                                   |

## 1. 기본 네비게이션

- `.` 현재 디렉토리 / `..` 부모 디렉토리 / `~` 홈 디렉토리
- 절대경로(`/`부터 시작) vs 상대경로(현재 위치 기준)

| 명령어 | 기능 |
| --- | --- |
| `pwd` | 현재 위치(경로) 출력 |
| `ls` | 현재 디렉토리 목록 |
| `cd 경로` | 디렉토리 이동 |
| `cd ~` | 홈 디렉토리로 이동 |
| `cd ..` | 부모 디렉토리로 이동 |

### 명령어 자체 탐색

| 명령어 | 기능 |
| --- | --- |
| `man 명령어` | 정식 매뉴얼 페이지 (자세함, `q`로 종료) |
| `명령어 --help` | 그 프로그램이 자체 제공하는 간략한 사용법 요약 |

- `which`와 `type`이 갈리는 대표 케이스 두 가지:
  - **alias**: `type`은 alias 내용을 그대로 보여주지만(`ls is aliased to '...'`), `which`는 alias를 못 잡아낼 수 있음
  - **빌트인 명령**: `cd`는 실행파일로 존재할 수 없음 — 프로세스별 작업 디렉토리(cwd)는 독립적이라, `cd`를 외부 프로그램(fork+exec)으로 만들면 자식 프로세스의 cwd만 바뀌고 부모 셸로 돌아오면 원상복구되어 무의미함. 그래서 `cd`는 반드시 셸 자체에 내장된 빌트인이어야 함
    - `which cd` → PATH에 실행파일이 없어 못 찾음
    - `type cd` → `cd is a shell builtin`으로 정확히 알려줌

### 히스토리 검색과 커서 이동

| 단축키 | 기능 |
| --- | --- |
| `Ctrl+R` | 히스토리 역방향 검색 (타이핑하면 매칭되는 과거 명령어를 실시간으로 찾음, 한 번 더 누르면 더 이전 매칭으로) |
| `Ctrl+A` | 커서를 줄 맨 앞으로 |
| `Ctrl+E` | 커서를 줄 맨 끝으로 |

## 2. 파일/디렉토리 조작

| 명령어 | 기능 |
| --- | --- |
| `mkdir 이름` | 디렉토리 생성 (`-p`: 중간 경로 없어도 한번에 생성) |
| `touch 이름` | 빈 파일 생성 (있으면 수정시각만 갱신) |
| `cp 원본 대상` | 복사 (`-r`: 디렉토리째로) |
| `mv 원본 대상` | 이동 또는 이름 변경 (같은 명령어) |
| `rm 이름` | 파일 삭제 |
| `rm -r 이름` | 디렉토리 삭제(안에 내용 있어도) |
| `rmdir 이름` | 빈 디렉토리만 삭제 (내용 있으면 에러) |

- `rm -rf`가 위험한 이유: `-r`(재귀 삭제) + `-f`(확인 없이 강제, 권한 무시)가 합쳐지면 오타 한 번으로 큰 디렉토리가 확인 절차 없이 영구 삭제됨. `rm`은 애초에 휴지통을 거치지 않아 복구 불가.

### 압축/아카이브

- **아카이빙**(여러 파일을 하나로 묶기, 용량 변화 없음)과 **압축**(용량 자체를 줄이기)은 서로 다른 작업
- `tar`: 아카이빙 도구(Tape ARchive). 압축은 별도 도구(gzip/bzip2/xz)에 맡기는 방식 — 유닉스 철학(한 가지 일을 잘하는 도구 조합)이 반영됨
- `zip`: 아카이빙+압축을 하나의 통합 포맷으로 처리 (윈도우 `.zip`이 익숙한 그 방식) — 대신 `tar`처럼 압축 알고리즘을 바꿔 끼우는 유연성은 없음

```bash
tar -czvf archive.tar.gz 폴더/     # 압축(-z: gzip) + 묶기(-c: create)
tar -xzvf archive.tar.gz           # 풀기(-x: extract)
```

- 옵션: `-c`(create) / `-x`(extract) / `-z`(gzip 압축·해제) / `-v`(verbose, 처리 파일 목록 출력) / `-f`(다음이 파일명이라는 표시, 보통 맨 뒤)
- `-z` 대신 `-j`(bzip2), `-J`(xz)로 압축 알고리즘 교체 가능

- vim: 저장/편집 기존에 숙지된 상태라 이번엔 별도 정리 생략

## 4. 권한과 소유권

### 정의
- `ls -l` 출력 맨 앞 10글자: 첫 글자는 파일 종류(`-` 파일 / `d` 디렉토리 / `l` 심볼릭 링크), 나머지 9글자는 3글자씩 끊어서 **소유자(owner) / 그룹(group) / 기타(others)** 권한.
  예시 : -rwxr-xr--
- 각 3글자는 `r`(read)/`w`(write)/`x`(execute) 고정 순서, 권한이 없으면 `-`.

### 원리
- `rwx`는 3개의 온/오프 스위치로 볼 수 있음: `r`=4, `w`=2, `x`=1 (2진수 자리값 100/010/001).
- 각 권한 값을 더해 한 자리 숫자로 표현 (예: `rwx`=7, `rw-`=6, `r-x`=5, `r--`=4, `---`=0).
- 소유자/그룹/기타 세 자리 숫자를 이어 붙인 게 `chmod`의 숫자 표기법 (예: `754`, `644`, `700`).

### 이유
- **최소 권한 원칙**: 필요한 만큼만 권한을 부여해야 보안상 안전. 개인용 스크립트는 `700`(소유자만 rwx), 공유/공개용은 `755`(소유자만 수정 가능, 나머지는 읽기+실행) 정도가 적절.
- `chmod 777`은 소유자/그룹/기타 전부에게 읽기/쓰기/실행을 다 허용하는 것이라, 같은 시스템의 아무나 파일을 수정·실행할 수 있게 되어 위험.

### 핵심 차이/포인트
- `chmod`: 권한(rwx) 자체를 변경. `chown`: 소유자/그룹을 변경.
- 본인이 소유자인 파일은 `sudo` 없이 `chmod` 가능. `chown`은 보통 관리자 권한(`sudo`)이 필요.
- 실행 권한 없는 스크립트는 `./script.sh` 실행 시 `Permission denied` 발생 → `chmod`로 실행 권한(`x`)을 부여해야 실행 가능.

### 명령어 예시
| 명령어 | 기능 |
| --- | --- |
| `ls -l` | 파일 권한/소유자/그룹 확인 |
| `chmod 755 file` | 소유자 rwx, 그룹/기타 r-x로 변경 |
| `chmod 700 file` | 소유자만 rwx, 나머지 권한 없음 |
| `chown user:group file` | 소유자와 그룹 동시 변경 |

## 5. 파이프와 리다이렉션

### 정의
- 모든 프로그램은 기본적으로 세 개의 통로(파일디스크립터)를 갖고 시작함: 
  입력 : **stdin**(표준입력, fd 0)  
  출력 : **stdout**(표준출력, fd 1) / **stderr**(표준에러, fd 2).
- 기본값은 stdin=키보드, stdout/stderr=터미널 화면. 리다이렉션·파이프는 이 연결을 다른 곳으로 바꾸는 것.
- `>` : stdout을 파일로 덮어쓰기(overwrite) / `>>` : stdout을 파일에 추가(append)
- `<` : 파일을 stdin으로 밀어넣기
- `|`(파이프) : 한 프로그램의 stdout을 다른 프로그램의 stdin에 직접 연결 (파일을 거치지 않음)
- `2>` : stderr만 파일로 리다이렉트 / `2>&1` : stderr를 "지금 stdout이 가리키는 곳"으로 연결

### 원리
- 셸은 명령어를 **실행(exec)하기 전에** 리다이렉션 기호들을 먼저 해석해서 자식 프로세스의 fd 테이블을 완성한다(`dup2`로 fd를 원하는 대상에 연결). 그 설정이 다 끝난 뒤에야 실제 프로그램이 실행된다.
- 리다이렉션이 명령줄 어디에 위치하든(앞이든 뒤든) 상관없이, 셸은 실행 전에 전부 걷어내 처리한다. 다만 리다이렉션이 여러 개면 그것들 **끼리는** 왼쪽→오른쪽 순서로 하나씩 fd가 설정된다.
- 그래서 `> both.txt 2>&1`은 
  ①stdout을 both.txt로 연결 → ②stderr를 "지금 stdout이 가리키는 곳(=both.txt)"으로 연결 → 결과: 둘 다 파일로.

- 반대로 `2>&1 > both2.txt`는 
  ①stderr를 "지금 stdout이 가리키는 곳(=아직 터미널)"으로 연결 → ②stdout만 파일로 변경 → 결과: stdout만 파일로, stderr는 터미널에 그대로 남음.

  - 프로그램(`ls` 등)은 이 모든 설정이 끝난 뒤에 태어나서(exec), 자기가 쓰는 fd 1/2가 파일인지 터미널인지 모른 채 그냥 정해진 번호에 쓰기만 한다.

### 핵심 차이/포인트
- `>`(덮어쓰기) vs `>>`(추가): `>`는 실행할 때마다 파일을 비우고 새로 씀, `>>`는 기존 내용 뒤에 이어붙임.
- `>`는 stdout만 리다이렉트, stderr는 그대로 터미널에 남음 → 에러까지 잡으려면 `2>` 또는 `2>&1` 필요.
- `2>&1`의 위치가 리다이렉션 순서를 바꾼다: `> file 2>&1`(stdout+stderr 모두 file로) ≠ `2>&1 > file`(stdout만 file로, stderr는 터미널).
- 파이프(`|`)는 두 프로그램을 stdout→stdin으로 직접 연결하는 것이고, 리다이렉션(`>`, `<`)은 프로그램과 파일을 연결하는 것 — 대상이 다름.

### 명령어 예시
| 명령어 | 기능 |
| --- | --- |
| `ls -l > result.txt` | 출력을 result.txt에 덮어쓰기 |
| `echo "text" >> note.txt` | 출력을 note.txt 끝에 추가 |
| `ls -l \| grep ".txt"` | ls 결과를 grep으로 넘겨 ".txt" 포함 줄만 필터링 |
| `sort < note.txt` | note.txt 내용을 sort의 stdin으로 입력 |
| `cmd 2> error.txt` | stderr만 error.txt로 리다이렉트 |
| `cmd > both.txt 2>&1` | stdout, stderr 모두 both.txt로 리다이렉트 |

## 6. 텍스트 처리 도구

### grep — 패턴으로 줄 찾기
- 기본형: `grep "패턴" 파일` → 파일을 줄 단위로 훑어 패턴이 포함된 줄만 출력

| 옵션   | 기능                  |
| ---- | ------------------- |
| `-i` | 대소문자 구분 안 함         |
| `-v` | 패턴 포함 안 된 줄만 (반대로)  |
| `-c` | 매칭된 줄 개수만 출력        |
| `-n` | 줄 번호도 같이 출력         |
| `-r` | 하위 디렉토리까지 재귀 검색     |
| `-e` | 패턴 여러 개 지정 (OR로 취급) |

- 정규식 없이 "여러 조건 다 만족하는 줄"을 찾고 싶을 때: grep을 파이프로 연결
  ```
  grep "login" app.log | grep "alice"
  ```
  첫 grep이 1차로 거르고, 그 결과를 두 번째 grep이 다시 좁히는 방식 — 순차적 AND.

### find — 파일 시스템에서 파일 자체를 찾기
- `grep`은 파일 **내용**(줄)을 뒤지고, `find`는 파일 **시스템**(파일/폴더 자체)을 뒤진다는 게 핵심 차이.
- 기본형: `find 경로 -조건`
  ```
  find . -name "*.c"      # 현재 위치 기준 .c 확장자 파일 찾기
  find . -name "*.c" | wc -l   # 개수 세기 (find + wc 조합)
  ```

### sort, uniq, wc — 정렬 / 중복 제거 / 개수 세기

| 명령어 | 기능 |
| --- | --- |
| `sort` | 줄 단위 정렬 (기본 사전순, `-n` 숫자순, `-r` 역순) |
| `uniq` | **연속된** 중복 줄만 제거 |
| `uniq -c` | 중복 제거하면서 각 줄이 몇 번 나왔는지 개수 출력 |
| `wc -l` | 줄 개수 세기 (`-w` 단어 수, `-c` 바이트 수) |

- `uniq`가 `sort`랑 세트로 다니는 이유: `uniq`는 **연속된** 중복만 잡아내기 때문에, 정렬 안 된 입력에서는 같은 값이 떨어져 있어 제대로 못 걸러냄. 그래서 항상 `sort` → `uniq` 순서로 씀.

### awk — 필드 단위로 잘라서 다루기
- 핵심 아이디어: 한 줄을 공백 기준으로 자동으로 여러 필드(`$1`, `$2`, `$3`...)로 쪼개줌.
- 예: `2026-07-10 09:01:03 INFO  user login: alice` → `$1`=날짜, `$2`=시간, `$3`=레벨, `$4`=`user`, `$5`=`login:`, `$6`=`alice`
- `awk '{print $N}'`: 각 줄의 N번째 필드만 출력

### 조합 예시: 로그에서 사용자별 로그인 횟수 세기
```
grep "login" app.log | awk '{print $6}' | sort | uniq -c
```
1. `grep "login"` → login 줄만 필터링
2. `awk '{print $6}'` → 각 줄에서 사용자 이름(마지막 필드)만 추출
3. `sort` → 같은 이름끼리 붙게 정렬
4. `uniq -c` → 이름별 등장 횟수 세기

실행 결과:
```
      2 alice
      1 bob
```

## 7. 프로세스 관리

### 프로세스란
- 프로세스: 실행 중인 프로그램의 인스턴스. 프로그램(디스크의 정적 파일) vs 프로세스(메모리에서 실행 중인 동적 실체)
- 같은 프로그램도 여러 번 실행하면 서로 다른 프로세스가 됨
- 커널은 PCB(Process Control Block)로 프로세스별 상태 관리 — PID, 실행 상태, 레지스터 값, 파일 디스크립터 테이블 등 포함

### 생성 과정: fork + exec
1. `fork()` — 셸 자신을 복제해 자식 프로세스 생성
2. `exec()` — 자식의 내용물을 실행할 프로그램으로 교체
3. 실행 종료 → 부모가 `wait()`로 종료 상태 수거 → 프롬프트 복귀
- 부모가 `wait()` 안 하면 자식은 좀비(`Z`) 상태로 PCB만 남음. 부모까지 죽으면 init(PID 1)이 입양 후 정리

### 상태 확인
- `ps aux`: 스냅샷(정적) / `top`, `htop`: 실시간 갱신, CPU·메모리 순 정렬 (`htop`이 개선판)
- `STAT` 컬럼
  - `R` Running
  - `S` Sleeping (인터럽트 가능한 대기)
  - `D` Uninterruptible sleep (디스크 I/O 대기)
  - `Z` Zombie
  - `T` Stopped
- 부가 문자(`s`=세션리더, `l`=멀티스레드, `+`=포그라운드 그룹)가 붙어 `Ss`, `Sl` 형태로 표시됨

### 포그라운드/백그라운드, job control
- 기본은 포그라운드 실행 (셸이 끝날 때까지 대기)
- 명령어 뒤 `&` → 백그라운드 실행, 셸은 바로 프롬프트 반환
- 셸이 백그라운드 작업마다 job 번호(`[1]`, `[2]`...) 부여 — PID와 별개, 세션 내 참조용
- `jobs` — 현재 세션의 백그라운드/정지 작업 목록 확인
- `Ctrl+Z` — `SIGTSTP` 전송, 일시정지(`T` 상태, 종료 아님)
- `bg %1` — 정지된 job 백그라운드에서 재개
- `fg %1` — job을 포그라운드로 전환

### 종료: kill vs kill -9
- `kill`(기본) — `SIGTERM`(15) 전송, 정중한 종료 요청. 프로세스가 캐치해서 cleanup(파일 저장, 락 해제 등) 후 스스로 종료 가능
- `kill -9` — `SIGKILL`(9) 전송, 캐치·무시·블록 불가. 커널이 즉시 강제 종료, cleanup 기회 없음
- 실무 순서: `kill` 먼저 시도 → 반응 없으면 최후 수단으로 `kill -9`

### nohup — 터미널 종료에도 생존
- 터미널 닫힘 → 셸에 `SIGHUP` 전달 → 셸이 자식들에게도 전달 → 일반 백그라운드(`&`)도 같이 죽음
- `nohup` — `SIGHUP` 무시하도록 감싸는 래퍼
- stdout 연결할 터미널이 없어지므로 출력은 자동으로 `nohup.out` 파일로 리다이렉트됨

### 명령어 예시
| 명령어 | 기능 |
| --- | --- |
| `ps aux` | 현재 떠있는 모든 프로세스 스냅샷 확인 |
| `top` / `htop` | 실시간 프로세스 모니터링 (`q`로 종료) |
| `sleep 300 &` | 명령어를 백그라운드로 실행 |
| `jobs` | 현재 셸 세션의 백그라운드/정지 작업 목록 확인 |
| `fg %1` | job 1번을 포그라운드로 전환 |
| `bg %1` | 정지된 job 1번을 백그라운드에서 재개 |
| `kill %1` | job 1번에 SIGTERM 전송 (정상 종료 요청) |
| `kill -9 %1` | job 1번에 SIGKILL 전송 (강제 종료) |
| `nohup sleep 300 &` | 터미널을 닫아도 살아있는 백그라운드 작업 실행 |

## 8. 셸 스크립팅

### 스크립트 기본 구조

```bash
#!/bin/bash
echo "안녕"
```

- 1번째 줄(**셔뱅**, `#!`)은 "이 파일을 어떤 인터프리터로 실행할지" 커널에게 알려주는 표시.
- `./script.sh`처럼 직접 실행할 때만 의미 있음: 커널이 `execve()`로 실행하려다 텍스트 형식을 인식 못 하면(`ENOEXEC`), 호출한 셸이 fallback으로 `/bin/sh`를 대신 써버림 → bash 전용 문법이 깨질 수 있음.
- `bash script.sh`처럼 인터프리터를 직접 명시해서 실행하면 셔뱅은 무시됨(이미 뭘로 실행할지 정해줬으니까).

### 변수와 따옴표

- 대입 시 `=` 양옆에 공백 있으면 안 됨(`name = "alice"`는 `name`이라는 명령어 실행 시도로 오해받음).
- 변수를 따옴표 없이 쓰면(`$name`) **워드 스플리팅** 발생 — `IFS`(기본: 공백/탭/줄바꿈) 기준으로 값을 다시 쪼갬. 따옴표로 감싸면(`"$name"`) 쪼개지지 않고 원본 그대로 보존.

```bash
name="alice    smith"
echo $name      # alice smith  (스페이스 1개로 뭉개짐 — 쪼개졌다가 합쳐짐)
echo "$name"    # alice    smith  (원본 스페이스 그대로 보존)
```

- 실무 규칙: **변수는 기본적으로 항상 큰따옴표로 감싸서 사용.**

### 위치 인자

`./greet.sh alice bob`로 실행했을 때:

| 변수 | 값 | 의미 |
| --- | --- | --- |
| `$0` | `./greet.sh` | 스크립트 자기 자신의 이름/경로 |
| `$1`, `$2` | `alice`, `bob` | 첫 번째, 두 번째 인자 |
| `$#` | `2` | 인자 개수 (`$0` 제외) |
| `$@` | `alice bob` | 전체 인자 (각각 개별 단어로 유지, `"$@"`로 쓰는 게 안전) |

```bash
for arg in "$@"; do
  echo "인자: $arg"
done
```

### export와 환경변수

- 그냥 `name="alice"` → 현재 셸 프로세스 내부 메모리에만 존재. 자식 프로세스는 모름.
- `export name="alice"` → "환경변수 블록"에 등록됨. `fork()`가 자식을 만들 때 이 블록을 복제해서 넘겨줌 → 자식이 `exec()`한 프로그램도 이 값을 물려받음.
- `.bashrc`에서 다른 프로그램이 알아야 할 값(`PATH`, `EDITOR` 등)엔 반드시 `export` 필요.

### .bashrc

- 홈 디렉토리(`~/.bashrc`)에 있고, **bash가 새로 시작될 때마다 자동으로 읽어서 실행**하는 설정 파일.
- alias, 환경변수, 프롬프트 설정 등을 여기에 넣어둠.

```bash
alias ll='ls -al'
export PATH="$HOME/bin:$PATH"
export EDITOR="vim"
```

### 제어문: if

```bash
if [ "$1" = "hello" ]; then
  echo "인사네"
else
  echo "인사 아니네"
fi
```

- `[`는 bash 문법이 아니라 **진짜 명령어**(`test`와 동일, `type [`로 확인 가능). 그래서 다른 명령어처럼 이름과 인자 사이에 공백이 꼭 필요함. 마지막 `]`도 그냥 마지막 인자일 뿐 — `[` 명령어가 스스로 마지막 인자가 `]`인지 검사함.
- 숫자 비교는 `<`/`>`를 안 쓰고 문자 옵션을 씀 — bash가 `<`/`>`를 리다이렉션 기호로 먼저 해석해버리기 때문.

| 연산자 | 의미 |
| --- | --- |
| `-eq` / `-ne` | equal / not equal |
| `-lt` / `-le` | less than / less than or equal |
| `-gt` / `-ge` | greater than / greater than or equal |

### 제어문: for와 글로빙

```bash
for file in *.txt; do
  echo "파일: $file"
done
```

- `*.txt`는 for문 실행 **전에** 셸이 먼저 실제 파일 목록으로 바꿔치기함(**글로빙**, 경로명 확장).
- 함정: 매칭되는 파일이 하나도 없으면, bash 기본 동작은 빈 목록이 아니라 **패턴 문자열 자체(`*.txt`)를 리터럴로 남김** → 루프가 0번이 아니라 1번 돌면서 `$file`에 `"*.txt"`라는 글자 그대로가 들어감.
- 이 특성을 역이용해서 "파일 없음" 케이스를 감지 가능:
  ```bash
  for file in *.txt; do
      if [ "$file" = "*.txt" ]; then
          continue   # 진짜 파일이 아니므로 건너뜀
      fi
      cp "$file" "backup_$file" || echo "복사 실패: $file"
  done
  ```
  (참고: `shopt -s nullglob` 옵션을 켜면 매칭 없을 때 진짜로 빈 목록이 되게 만들 수도 있음)

### 제어문: while

```bash
count=1
while [ $count -le 5 ]; do
  echo "count: $count"
  count=$((count + 1))
done
```

- `$((...))`: 산술 연산 문법.

### 함수와 반환값

```bash
greet() {
  echo "안녕, $1"
}
greet "alice"
```

- 함수 안의 `$1`, `$2`, `$@`는 스크립트 전체 인자가 아니라 **그 함수를 호출할 때 넘긴 인자**로 새로 바뀜(shadowing).
- `return`은 실제 값 반환이 아니라 **0~255 사이의 정수(종료 코드)만** 반환 가능 → 계산값이 256을 넘으면 나머지 연산으로 잘못된 값이 나옴 (예: `return 300` → 실제론 `44`).
- 진짜 계산값을 돌려받고 싶으면 `echo` + 커맨드 치환 사용:
  ```bash
  add() {
    echo $(($1 + $2))
  }
  result=$(add 3 4)
  ```

### 종료 코드(`$?`)와 `&&`/`||`

- 모든 명령어/함수는 끝나면 종료 코드를 남김 — `$?`로 확인. **0=성공, 1~255=실패**.
- `A && B` : A가 성공(0)해야 B 실행
- `A || B` : A가 실패(0 아님)해야 B 실행

```bash
mkdir backup && echo "생성 성공"
cp file.txt backup/ || echo "복사 실패!"
```

### 실습: 파일 일괄 백업 스크립트

```bash
#!/bin/bash
echo "복사본 생성"

for backup in *.txt; do
    if [ "$backup" = "*.txt" ]; then
        continue
    fi
    cp "$backup" backup_"$backup" || echo "복사 실패"
done
```

## 9. 네트워크 & 원격

### 포트(Port) 개념

- IP 주소는 컴퓨터(호스트) 자체를 찾아가는 주소, 포트는 그 컴퓨터 안에서 **어떤 프로그램에게 전달할지** 구분하는 번호(0~65535)
- 같은 컴퓨터에서 여러 프로그램이 동시에 네트워크 통신을 하기 때문에 IP만으로는 목적지 프로그램을 특정할 수 없음 → 포트로 구분
- 잘 알려진 포트: `22`(SSH), `80`(HTTP), `443`(HTTPS)

### SSH — 원격 셸 접속

- 원격 컴퓨터에 접속해서 마치 로컬처럼 명령어를 실행할 수 있게 해주는 프로토콜, 통신 내용 암호화
- 인증 방식 두 가지: 비밀번호 인증 / SSH 키 인증

**키 인증 원리 (challenge-response)**
- 개인키 값 자체는 네트워크로 전송되지 않음 (비밀번호와의 차이점)
  1. 서버가 랜덤한 challenge를 만들어 클라이언트에 보냄
  2. 클라이언트가 개인키로 그 challenge에 서명해서 반환
  3. 서버가 등록된 공개키로 서명을 검증 — "그 개인키를 가진 사람만 만들 수 있는 서명"임을 확인
- 즉 "개인키를 갖고 있다는 증거"만 오가고 개인키 값 자체는 절대 노출 안 됨 → 도청해도 개인키 유추 불가

**패스프레이즈(passphrase)**
- 개인키 파일은 기본적으로 디스크에 평문 저장 → 파일 자체가 유출되면(도난 등) 그대로 인증 통과 가능
- 패스프레이즈는 개인키 파일 자체를 암호화하는 추가 비밀번호 → 파일을 훔쳐도 패스프레이즈 없으면 내부 키를 꺼내 쓸 수 없음
- 이중 방어 구조: 키 쌍 구조(네트워크 도청 방어) + 패스프레이즈(키 파일 도난 방어)

**키 쌍은 두 종류 — 사용자 인증용 vs 호스트 키**
- 키는 항상 개인키+공개키가 **한 쌍으로 같이 생성됨** (`ssh-keygen` 실행 시 `id_rsa`(개인)/`id_rsa.pub`(공개) 동시 생성). 키를 생성한 쪽이 둘 다 로컬에 갖고 있다가 **공개키만 복사해서 상대에게 배포**하는 구조
- 용도가 다른 두 키 쌍이 각자 반대 방향으로 존재함

| | 개인키 보관 | 공개키 보관 |
| --- | --- | --- |
| **사용자 인증 키 쌍** (내가 나임을 증명) | 클라이언트 | 서버 (`~/.ssh/authorized_keys`) |
| **호스트 키 쌍** (서버가 자기 자신임을 증명) | 서버 (`/etc/ssh/ssh_host_*_key`) | 클라이언트 (`~/.ssh/known_hosts`) |

- 접속 한 번에 양방향 신원 확인이 다 일어남: 서버는 호스트 키로 "나 진짜 서버 맞아" 증명, 클라이언트는 사용자 키로 "나 진짜 그 사용자 맞아" 증명 — 서로 같은 방식(개인키로 서명 → 공개키로 검증)을 대칭적으로 사용

**TOFU (Trust On First Use)**
- 호스트 키의 문제: 공개키 자체만으로는 "이게 원래 신뢰할 만한 서버 것인지"는 검증 불가 (서명 검증은 "지금 상대가 그 개인키를 가졌다"만 증명, 원래 신뢰할 만한 서버인지는 별개)
- 그래서 SSH는 접속 시 "이 서버 처음 보는데 이 공개키를 믿고 진행할까?" 라고 물어봄 → 처음 접속할 때는 그냥 믿고 넘어가서 `known_hosts`에 저장, 이후 접속부터는 저장된 값과 비교해서 값이 바뀌면 강한 경고를 띄우는 방식
- 완벽한 검증은 아님 — 최초 접속 순간에 이미 중간자가 끼어있었다면 가짜 키를 정상으로 신뢰해버릴 수 있음. 보안이 중요한 환경에서는 out-of-band(전화 등 별도 채널)로 지문값을 대조하기도 함
- localhost 접속 시에도 동일한 호스트 키 검증 메시지가 뜨는데, loopback이라 중간자가 낄 네트워크 경로 자체가 없어 실질적 위협은 거의 없음 — 다만 SSH 프로토콜이 접속 대상을 구분해서 동작하지 않기 때문에 동일 로직이 항상 적용됨

### scp / rsync — 원격 파일 전송

- 새로운 암호화 체계를 만드는 게 아니라, SSH가 이미 만들어둔 인증+암호화 통로를 그대로 재활용해서 파일 데이터를 실어 보냄
- `rsync`는 `scp`와 달리 파일 전체를 매번 복사하지 않고, 체크섬으로 비교해서 **바뀐 부분(delta)만** 전송 → 반복 동기화(백업, 배포 등)에 효율적

| 명령어 | 기능 |
| --- | --- |
| `scp 파일 user@호스트:경로` | 로컬 → 원격 복사 |
| `scp user@호스트:경로 .` | 원격 → 로컬 복사 |
| `rsync -avz 로컬경로/ user@호스트:원격경로/` | 변경분만 동기화 |

- `rsync` 옵션: `-a`(권한/링크/타임스탬프 보존 + 재귀), `-v`(진행상황 출력), `-z`(전송 중 압축)
- 경로 끝 `/` 유무로 동작 차이: `경로/`(슬래시 있음) → 폴더 안 내용물만 복사, `경로`(슬래시 없음) → 폴더 자체를 대상 밑에 복사

### curl / wget — HTTP(S) 요청

- SSH 계열과 달리 주로 HTTP/HTTPS로 통신 (URL 스킴에 따라 자동으로 80번/443번 포트 사용)
- `wget`: URL 파일 다운로드에 특화
- `curl`: 다운로드뿐 아니라 API 요청(POST, 헤더 커스터마이징 등)에 유연

```bash
curl https://example.com/api/data
wget https://example.com/file.zip
```

## 10. 고급 활용

### 환경변수와 PATH

- **환경변수**: 셸 메모리에 있는 키=값 저장소 중 `export`로 등록된 것 — 일반 변수와 달리 자식 프로세스에게도 상속됨
- **PATH**: 셸이 명령어 실행 시 실행파일을 찾아 뒤지는 디렉토리 목록(`:`로 구분). **왼쪽부터 순서대로** 탐색해서 제일 먼저 매치되는 걸 실행
- 실무 활용: `pyenv`/`nvm` 같은 버전 관리 도구는 새로운 실행 메커니즘을 만드는 게 아니라, `PATH` 맨 앞에 자기 관리 디렉토리를 끼워 넣어서 시스템 기본 버전보다 먼저 잡히게 하는 방식

```bash
echo $PATH
export PATH="$HOME/bin:$PATH"   # 원하는 경로를 앞에 붙여 우선 실행되게 함
```

### alias

- 명령어(조합)에 별명을 붙이는 기능. 등록만 하면 그 이름을 칠 때 실제 명령어로 치환됨

```bash
alias ll='ls -al'
alias gs='git status'
```

- **명령어 해석 순서**: alias → 셸 빌트인 → PATH 탐색. alias가 걸리면 그 자리에서 바로 치환되고 PATH 탐색 자체를 안 함
- `type`은 alias 내용까지 그대로 보여주지만(`ls is aliased to 'rm -rf ~'`), `which`는 셸에 따라 alias를 못 잡아낼 수 있음 → "이 명령어가 진짜 뭘 실행하는지" 의심될 땐 `type`이 더 믿을만함

### tmux — 터미널 멀티플렉서

**필요한 이유**
- SSH 연결이 끊기면 원격 셸에 `SIGHUP`이 전달되고, 셸이 이를 자식 프로세스들에게 전파 → 돌아가던 작업이 다 같이 죽음
- tmux는 **서버-클라이언트 구조**로 이 문제를 해결: `tmux new -s work`를 실행하면 백그라운드에 tmux 서버 프로세스가 뜨고, 보고 있는 터미널 화면은 그 서버에 붙은 클라이언트일 뿐
- `Ctrl+b d`(detach)는 클라이언트 연결만 끊는 것 — 서버와 그 안의 프로세스들은 특정 pty에 안 묶여있어서 독립적으로 계속 실행됨 (얼려서 보존되는 게 아니라 실시간으로 계속 진행 중인 상태)

**계층 구조**
```
session (전체 작업 단위, detach/attach 기준)
 └─ window (탭 개념)
     └─ pane (화면 분할)
```

**명령어**

| 명령/키 | 기능 |
| --- | --- |
| `tmux new -s work` | "work" 세션 시작 |
| `tmux ls` | 세션 목록 확인 |
| `tmux attach -t work` | 세션 재접속 |
| `Ctrl+b d` | 세션에서 분리 (detach) |
| `Ctrl+b %` | pane 세로 분할 |
| `Ctrl+b "` | pane 가로 분할 |
| `Ctrl+b 방향키` | pane 간 이동 |
| `Ctrl+b z` | pane 전체화면 토글 (zoom) |
| `Ctrl+b c` | 새 window 생성 |
| `Ctrl+b n` / `p` | 다음/이전 window 이동 |
| `Ctrl+b 숫자` | 해당 번호 window로 이동 |

- `Ctrl+b`는 prefix 키 — tmux 단축키는 항상 이걸 먼저 누른 뒤 다음 키를 눌러야 tmux가 인식함

**로컬 환경에서의 tmux**
- 로컬에서 "화면 여러 개 놓고 보기"만이 목적이면 요즘 터미널 앱(iTerm2, Windows Terminal 등) 자체 분할 기능으로 충분 — tmux 주 사용처는 원격 서버 작업
- 다만 터미널 창을 그냥 닫는 것도 tmux 입장에선 사실상 detach와 동일한 효과(클라이언트만 죽고 서버는 생존) → 로컬에서도 터미널 앱이 죽거나 실수로 창을 닫아도 세션은 보존됨
- 단, 이건 "컴퓨터 자체가 켜져있는 동안"이라는 전제가 붙음 — 컴퓨터를 끄면 tmux 서버도 같이 죽음 (원격 서버 케이스와의 차이점: 원격은 로컬 노트북을 꺼도 서버가 딴 컴퓨터라 안 죽음)

### dotfiles 관리

- 홈 디렉토리의 `.`으로 시작하는 설정 파일들(`.bashrc`, `.vimrc`, `.gitconfig` 등)을 통칭
- 새 컴퓨터 세팅할 때마다 반복 작업/누락을 막기 위해 git 레포로 관리하고, 심볼릭 링크로 홈 디렉토리에 연결하는 방식이 일반적

```bash
git clone https://github.com/계정/dotfiles.git ~/dotfiles
ln -s ~/dotfiles/.bashrc ~/.bashrc
```

- `cp` 대신 `ln -s`를 쓰는 이유: `cp`는 원본과 물리적으로 분리된 복사본이라 홈 디렉토리에서 수정해도 git 레포 쪽엔 반영 안 되고 버전이 갈라짐(drift). 심볼릭 링크는 물리적으로 같은 파일이라 어디서 고치든 git이 바로 변경 감지 가능
- **yadm**: 이 방식의 발전형. 심볼릭 링크를 안 쓰고 "bare git 저장소" 트릭으로 `$HOME` 자체를 git 작업 디렉토리로 취급 → 파일이 원래 자리에 그대로 있으면서 git 추적도 됨. 머신별/OS별 분기, 민감 파일 암호화 등 부가 기능도 지원

## 11. 시스템 관리 기초

### 패키지 관리자 (apt)

- `apt update`: 저장소(`/etc/apt/sources.list`)에서 패키지 인덱스(메타데이터 — 어떤 패키지가 있고 최신 버전이 몇인지)를 로컬에 갱신. 실제 프로그램 파일은 아직 안 받음
- `apt install`: 로컬 인덱스를 보고 의존성을 계산한 뒤, 실제 `.deb` 파일을 저장소에서 다운로드해 설치
- `update` 없이 `install`하면 오래된 인덱스 정보로 설치를 시도해서 실패하거나 최신 버전이 아닌 걸 설치할 수 있음

```bash
sudo apt update              # 패키지 인덱스 최신화
sudo apt install <패키지명>   # 설치
sudo apt remove <패키지명>    # 삭제 (설정파일은 남김)
sudo apt purge <패키지명>     # 삭제 + 설정파일까지 제거
sudo apt upgrade             # 설치된 패키지 업데이트
apt list --upgradable        # 업데이트 가능한 패키지 목록만 확인
```

### df vs du

- `df`: **파일시스템(마운트된 디스크/파티션) 전체** 관점에서 사용량/여유공간 확인
- `du`: **특정 경로 하나**를 지정해서 그 안의 파일들만 합산
- 둘이 어긋나 보이는 이유: `du ~`는 홈 디렉토리 안만 더한 값이지만, `df`가 보는 파일시스템에는 `/usr`, `/var`, `/etc`, 다른 계정, 시스템 로그 등 훨씬 더 많은 게 포함되어 있음
- 심화 edge case: 프로세스가 파일을 열어둔 채로 그 파일이 삭제되면, `du`는 (디렉토리 목록에서 사라졌으니) 못 잡아내지만 `df`는 프로세스가 파일을 닫기 전까지 그 디스크 블록을 계속 "사용 중"으로 계산 — 로그 파일 로테이션 시 흔히 겪는 문제

```bash
df -h                                          # 파일시스템 전체 용량
du -sh 경로                                    # 특정 경로 전체 용량 합산
du -h ~ --max-depth=1 | sort -rh | head        # 홈 디렉토리 내 용량 큰 폴더 순 정렬
```
