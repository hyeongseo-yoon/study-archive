## 0. 터미널의 기초 개념

### 정의

- **터미널(Terminal)**: 사용자와 프로그램(주로 셸) 사이를 이어주는 **입출력 인터페이스**. 키 입력을 받아 셸에 전달하고, 셸이 돌려주는 결과를 화면에 표시하는 역할만 함 — 명령어 자체를 해석하지는 않음. 그냥 "창문" 역할.
- **셸(Shell, bash/zsh 등)**: 터미널 안에서 실제로 명령어를 해석하고 실행하는 프로그램. 창문 뒤에서 일하는 "안내원/일꾼" 역할.
- 터미널과 셸은 서로 다른 개념이며, 셸에는 bash/zsh 등 여러 종류가 있고 문법이 조금씩 다르다.
- 현재 사용 중인 셸 확인: `echo $SHELL`

### 원리

- GUI는 마우스로 아이콘/폴더를 클릭해서 조작하는 방식.
- CLI(터미널)는 키보드로 명령어를 쳐서 조작하는 방식.
- 예: 폴더 100개 안의 파일을 각각 지워야 할 때, GUI는 더블클릭 → 우클릭 → 삭제를 100번 반복해야 하지만, CLI는 명령어 한 줄로 한 번에 처리 가능.
- **명령 처리 흐름**: 사용자 → (터미널이 입력을 받아 전달) → 셸이 명령어 해석 → 실제 프로그램 실행 → 결과가 다시 터미널을 통해 출력됨. 즉 터미널=입출력 인터페이스, 셸=해석/실행 담당이며 이 둘은 별개의 프로그램.

### 이유 (왜 CLI를 쓰는가)

- **자동화·반복**: 명령어는 정해진 절차(레시피)라서 여러 번 반복해도 항상 같은 동작을 보장.
- **재현성**: 마우스로 한 작업은 기록이 남지 않지만, 명령어는 그대로 복사해 남에게 전달하거나 스크립트로 저장해두면 언제든 동일하게 재현 가능.

###  명령어 예시

| 명령어           | 기능                                |
| ------------- | --------------------------------- |
| `echo hello`  | "hello" 문자열을 그대로 화면에 출력           |
| `whoami`      | 현재 로그인한 사용자 이름(계정명) 출력            |
| `date`        | 현재 날짜/시간 출력                       |
| `echo $SHELL` | 현재 사용 중인 셸 경로 확인 (예: `/bin/bash`) |
|               |                                   |

## 2. 파일/디렉토리 조작

| 명령어 | 기능 |
| --- | --- |
| `mkdir 이름` | 디렉토리 생성 (`-p`: 중간 경로 없어도 한번에 생성) |
| `touch 이름` | 빈 파일 생성 (있으면 수정시각만 갱신) |
| `cp 원본 대상` | 복사 (`-r`: 디렉토리째로) |
| `mv 원본 대상` | 이동 또는 이름 변경 (같은 명령어) |
| `rm 이름` | 파일 삭제 |
| `rm -r 이름` | 디렉토리 삭제(안에 내용 있어도) |
| `rmdir 이름` | 빈 디렉토리만 삭제 (내용 있으면 에러) |

- `rm -rf`가 위험한 이유: `-r`(재귀 삭제) + `-f`(확인 없이 강제, 권한 무시)가 합쳐지면 오타 한 번으로 큰 디렉토리가 확인 절차 없이 영구 삭제됨. `rm`은 애초에 휴지통을 거치지 않아 복구 불가.

- `cat`은 파일 전체를 한 번에 쏟아내서 큰 파일엔 부적합 → `less`(화면 단위) 또는 `head`/`tail`(일부만)로 대체
- vim: 저장/편집 기존에 숙지된 상태라 이번엔 별도 정리 생략

## 4. 권한과 소유권

### 정의
- `ls -l` 출력 맨 앞 10글자: 첫 글자는 파일 종류(`-` 파일 / `d` 디렉토리 / `l` 심볼릭 링크), 나머지 9글자는 3글자씩 끊어서 **소유자(owner) / 그룹(group) / 기타(others)** 권한.
  예시 : -rwxr-xr--
- 각 3글자는 `r`(read)/`w`(write)/`x`(execute) 고정 순서, 권한이 없으면 `-`.

### 원리
- `rwx`는 3개의 온/오프 스위치로 볼 수 있음: `r`=4, `w`=2, `x`=1 (2진수 자리값 100/010/001).
- 각 권한 값을 더해 한 자리 숫자로 표현 (예: `rwx`=7, `rw-`=6, `r-x`=5, `r--`=4, `---`=0).
- 소유자/그룹/기타 세 자리 숫자를 이어 붙인 게 `chmod`의 숫자 표기법 (예: `754`, `644`, `700`).

### 이유
- **최소 권한 원칙**: 필요한 만큼만 권한을 부여해야 보안상 안전. 개인용 스크립트는 `700`(소유자만 rwx), 공유/공개용은 `755`(소유자만 수정 가능, 나머지는 읽기+실행) 정도가 적절.
- `chmod 777`은 소유자/그룹/기타 전부에게 읽기/쓰기/실행을 다 허용하는 것이라, 같은 시스템의 아무나 파일을 수정·실행할 수 있게 되어 위험.

### 핵심 차이/포인트
- `chmod`: 권한(rwx) 자체를 변경. `chown`: 소유자/그룹을 변경.
- 본인이 소유자인 파일은 `sudo` 없이 `chmod` 가능. `chown`은 보통 관리자 권한(`sudo`)이 필요.
- 실행 권한 없는 스크립트는 `./script.sh` 실행 시 `Permission denied` 발생 → `chmod`로 실행 권한(`x`)을 부여해야 실행 가능.

### 명령어 예시
| 명령어 | 기능 |
| --- | --- |
| `ls -l` | 파일 권한/소유자/그룹 확인 |
| `chmod 755 file` | 소유자 rwx, 그룹/기타 r-x로 변경 |
| `chmod 700 file` | 소유자만 rwx, 나머지 권한 없음 |
| `chown user:group file` | 소유자와 그룹 동시 변경 |

## 5. 파이프와 리다이렉션

### 정의
- 모든 프로그램은 기본적으로 세 개의 통로(파일디스크립터)를 갖고 시작함: 
  입력 : **stdin**(표준입력, fd 0)  
  출력 : **stdout**(표준출력, fd 1) / **stderr**(표준에러, fd 2).
- 기본값은 stdin=키보드, stdout/stderr=터미널 화면. 리다이렉션·파이프는 이 연결을 다른 곳으로 바꾸는 것.
- `>` : stdout을 파일로 덮어쓰기(overwrite) / `>>` : stdout을 파일에 추가(append)
- `<` : 파일을 stdin으로 밀어넣기
- `|`(파이프) : 한 프로그램의 stdout을 다른 프로그램의 stdin에 직접 연결 (파일을 거치지 않음)
- `2>` : stderr만 파일로 리다이렉트 / `2>&1` : stderr를 "지금 stdout이 가리키는 곳"으로 연결

### 원리
- 셸은 명령어를 **실행(exec)하기 전에** 리다이렉션 기호들을 먼저 해석해서 자식 프로세스의 fd 테이블을 완성한다(`dup2`로 fd를 원하는 대상에 연결). 그 설정이 다 끝난 뒤에야 실제 프로그램이 실행된다.
- 리다이렉션이 명령줄 어디에 위치하든(앞이든 뒤든) 상관없이, 셸은 실행 전에 전부 걷어내 처리한다. 다만 리다이렉션이 여러 개면 그것들 **끼리는** 왼쪽→오른쪽 순서로 하나씩 fd가 설정된다.
- 그래서 `> both.txt 2>&1`은 
  ①stdout을 both.txt로 연결 → ②stderr를 "지금 stdout이 가리키는 곳(=both.txt)"으로 연결 → 결과: 둘 다 파일로.

- 반대로 `2>&1 > both2.txt`는 
  ①stderr를 "지금 stdout이 가리키는 곳(=아직 터미널)"으로 연결 → ②stdout만 파일로 변경 → 결과: stdout만 파일로, stderr는 터미널에 그대로 남음.

  - 프로그램(`ls` 등)은 이 모든 설정이 끝난 뒤에 태어나서(exec), 자기가 쓰는 fd 1/2가 파일인지 터미널인지 모른 채 그냥 정해진 번호에 쓰기만 한다.

### 핵심 차이/포인트
- `>`(덮어쓰기) vs `>>`(추가): `>`는 실행할 때마다 파일을 비우고 새로 씀, `>>`는 기존 내용 뒤에 이어붙임.
- `>`는 stdout만 리다이렉트, stderr는 그대로 터미널에 남음 → 에러까지 잡으려면 `2>` 또는 `2>&1` 필요.
- `2>&1`의 위치가 리다이렉션 순서를 바꾼다: `> file 2>&1`(stdout+stderr 모두 file로) ≠ `2>&1 > file`(stdout만 file로, stderr는 터미널).
- 파이프(`|`)는 두 프로그램을 stdout→stdin으로 직접 연결하는 것이고, 리다이렉션(`>`, `<`)은 프로그램과 파일을 연결하는 것 — 대상이 다름.

### 명령어 예시
| 명령어 | 기능 |
| --- | --- |
| `ls -l > result.txt` | 출력을 result.txt에 덮어쓰기 |
| `echo "text" >> note.txt` | 출력을 note.txt 끝에 추가 |
| `ls -l \| grep ".txt"` | ls 결과를 grep으로 넘겨 ".txt" 포함 줄만 필터링 |
| `sort < note.txt` | note.txt 내용을 sort의 stdin으로 입력 |
| `cmd 2> error.txt` | stderr만 error.txt로 리다이렉트 |
| `cmd > both.txt 2>&1` | stdout, stderr 모두 both.txt로 리다이렉트 |

## 6. 텍스트 처리 도구

### grep — 패턴으로 줄 찾기
- 기본형: `grep "패턴" 파일` → 파일을 줄 단위로 훑어 패턴이 포함된 줄만 출력

| 옵션   | 기능                  |
| ---- | ------------------- |
| `-i` | 대소문자 구분 안 함         |
| `-v` | 패턴 포함 안 된 줄만 (반대로)  |
| `-c` | 매칭된 줄 개수만 출력        |
| `-n` | 줄 번호도 같이 출력         |
| `-r` | 하위 디렉토리까지 재귀 검색     |
| `-e` | 패턴 여러 개 지정 (OR로 취급) |

- 정규식 없이 "여러 조건 다 만족하는 줄"을 찾고 싶을 때: grep을 파이프로 연결
  ```
  grep "login" app.log | grep "alice"
  ```
  첫 grep이 1차로 거르고, 그 결과를 두 번째 grep이 다시 좁히는 방식 — 순차적 AND.

### find — 파일 시스템에서 파일 자체를 찾기
- `grep`은 파일 **내용**(줄)을 뒤지고, `find`는 파일 **시스템**(파일/폴더 자체)을 뒤진다는 게 핵심 차이.
- 기본형: `find 경로 -조건`
  ```
  find . -name "*.c"      # 현재 위치 기준 .c 확장자 파일 찾기
  find . -name "*.c" | wc -l   # 개수 세기 (find + wc 조합)
  ```

### sort, uniq, wc — 정렬 / 중복 제거 / 개수 세기

| 명령어 | 기능 |
| --- | --- |
| `sort` | 줄 단위 정렬 (기본 사전순, `-n` 숫자순, `-r` 역순) |
| `uniq` | **연속된** 중복 줄만 제거 |
| `uniq -c` | 중복 제거하면서 각 줄이 몇 번 나왔는지 개수 출력 |
| `wc -l` | 줄 개수 세기 (`-w` 단어 수, `-c` 바이트 수) |

- `uniq`가 `sort`랑 세트로 다니는 이유: `uniq`는 **연속된** 중복만 잡아내기 때문에, 정렬 안 된 입력에서는 같은 값이 떨어져 있어 제대로 못 걸러냄. 그래서 항상 `sort` → `uniq` 순서로 씀.

### awk — 필드 단위로 잘라서 다루기
- 핵심 아이디어: 한 줄을 공백 기준으로 자동으로 여러 필드(`$1`, `$2`, `$3`...)로 쪼개줌.
- 예: `2026-07-10 09:01:03 INFO  user login: alice` → `$1`=날짜, `$2`=시간, `$3`=레벨, `$4`=`user`, `$5`=`login:`, `$6`=`alice`
- `awk '{print $N}'`: 각 줄의 N번째 필드만 출력

### 조합 예시: 로그에서 사용자별 로그인 횟수 세기
```
grep "login" app.log | awk '{print $6}' | sort | uniq -c
```
1. `grep "login"` → login 줄만 필터링
2. `awk '{print $6}'` → 각 줄에서 사용자 이름(마지막 필드)만 추출
3. `sort` → 같은 이름끼리 붙게 정렬
4. `uniq -c` → 이름별 등장 횟수 세기

실행 결과:
```
      2 alice
      1 bob
```

## 7. 프로세스 관리

### 프로세스란
- 프로세스: 실행 중인 프로그램의 인스턴스. 프로그램(디스크의 정적 파일) vs 프로세스(메모리에서 실행 중인 동적 실체)
- 같은 프로그램도 여러 번 실행하면 서로 다른 프로세스가 됨
- 커널은 PCB(Process Control Block)로 프로세스별 상태 관리 — PID, 실행 상태, 레지스터 값, 파일 디스크립터 테이블 등 포함

### 생성 과정: fork + exec
1. `fork()` — 셸 자신을 복제해 자식 프로세스 생성
2. `exec()` — 자식의 내용물을 실행할 프로그램으로 교체
3. 실행 종료 → 부모가 `wait()`로 종료 상태 수거 → 프롬프트 복귀
- 부모가 `wait()` 안 하면 자식은 좀비(`Z`) 상태로 PCB만 남음. 부모까지 죽으면 init(PID 1)이 입양 후 정리

### 상태 확인
- `ps aux`: 스냅샷(정적) / `top`, `htop`: 실시간 갱신, CPU·메모리 순 정렬 (`htop`이 개선판)
- `STAT` 컬럼
  - `R` Running
  - `S` Sleeping (인터럽트 가능한 대기)
  - `D` Uninterruptible sleep (디스크 I/O 대기)
  - `Z` Zombie
  - `T` Stopped
- 부가 문자(`s`=세션리더, `l`=멀티스레드, `+`=포그라운드 그룹)가 붙어 `Ss`, `Sl` 형태로 표시됨

### 포그라운드/백그라운드, job control
- 기본은 포그라운드 실행 (셸이 끝날 때까지 대기)
- 명령어 뒤 `&` → 백그라운드 실행, 셸은 바로 프롬프트 반환
- 셸이 백그라운드 작업마다 job 번호(`[1]`, `[2]`...) 부여 — PID와 별개, 세션 내 참조용
- `jobs` — 현재 세션의 백그라운드/정지 작업 목록 확인
- `Ctrl+Z` — `SIGTSTP` 전송, 일시정지(`T` 상태, 종료 아님)
- `bg %1` — 정지된 job 백그라운드에서 재개
- `fg %1` — job을 포그라운드로 전환

### 종료: kill vs kill -9
- `kill`(기본) — `SIGTERM`(15) 전송, 정중한 종료 요청. 프로세스가 캐치해서 cleanup(파일 저장, 락 해제 등) 후 스스로 종료 가능
- `kill -9` — `SIGKILL`(9) 전송, 캐치·무시·블록 불가. 커널이 즉시 강제 종료, cleanup 기회 없음
- 실무 순서: `kill` 먼저 시도 → 반응 없으면 최후 수단으로 `kill -9`

### nohup — 터미널 종료에도 생존
- 터미널 닫힘 → 셸에 `SIGHUP` 전달 → 셸이 자식들에게도 전달 → 일반 백그라운드(`&`)도 같이 죽음
- `nohup` — `SIGHUP` 무시하도록 감싸는 래퍼
- stdout 연결할 터미널이 없어지므로 출력은 자동으로 `nohup.out` 파일로 리다이렉트됨

### 명령어 예시
| 명령어 | 기능 |
| --- | --- |
| `ps aux` | 현재 떠있는 모든 프로세스 스냅샷 확인 |
| `top` / `htop` | 실시간 프로세스 모니터링 (`q`로 종료) |
| `sleep 300 &` | 명령어를 백그라운드로 실행 |
| `jobs` | 현재 셸 세션의 백그라운드/정지 작업 목록 확인 |
| `fg %1` | job 1번을 포그라운드로 전환 |
| `bg %1` | 정지된 job 1번을 백그라운드에서 재개 |
| `kill %1` | job 1번에 SIGTERM 전송 (정상 종료 요청) |
| `kill -9 %1` | job 1번에 SIGKILL 전송 (강제 종료) |
| `nohup sleep 300 &` | 터미널을 닫아도 살아있는 백그라운드 작업 실행 |
